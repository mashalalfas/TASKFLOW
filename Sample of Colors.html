⚠️
RX 6400 reality check: The RX 6400 has only 4GB VRAM. You're limited to 9B models at 4-bit quantization (~4.5GB, partial CPU offload) or smaller. The 9B will work but may run partly on CPU RAM. Expect ~3–8 tokens/second depending on offload.
01

AMD GPU / ROCm Setup

The RX 6400 is RDNA2 (gfx1017) but is not officially in ROCm's supported list. We use HSA_OVERRIDE_GFX_VERSION to force it to use a compatible target. Ollama now ships with ROCm 7.

Install ROCm Drivers

Install AMD ROCm via amdgpu-installbash
# Add AMD's repo (Ubuntu/Debian)
sudo apt update
sudo apt install -y wget gnupg

# Download and run AMD's installer tool
wget https://repo.radeon.com/amdgpu-install/latest/ubuntu/jammy/amdgpu-install_6.3.60300-1_all.deb
sudo apt install -y ./amdgpu-install_6.3.60300-1_all.deb

# Install ROCm + amdgpu driver
sudo amdgpu-install --usecase=rocm

# Add your user to the render and video groups
sudo usermod -aG render,video $LOGNAME

# Reboot!
sudo reboot

Verify GPU is Detected

Verify after rebootbash
rocminfo | grep "gfx\|Name"
# Should show: gfx1017 (or similar) for RX 6400

rocm-smi
# Should show your GPU with temperature, VRAM usage etc.

Set the GFX Override (Critical!)

Add this permanently to your shell config so Ollama always picks it up:

~/.bashrc or ~/.zshrcbash
# AMD RX 6400 — force ROCm to use gfx1030 target
export HSA_OVERRIDE_GFX_VERSION=10.3.0

# Optional: enable flash attention for better performance
export OLLAMA_FLASH_ATTENTION=1

# Reload
source ~/.bashrc
💡
If ROCm doesn't cooperate at all, Ollama will silently fall back to CPU. That's fine — it still works, just slower (~1–3 tok/s). You can always confirm GPU usage with rocm-smi while the model is running.
02

Install & Configure Ollama

Install

Install latest Ollama (v0.17+ required for Qwen3.5)bash
# Install / update to latest version
curl -fsSL https://ollama.com/install.sh | sh

# Verify version (need v0.17+ for Qwen3.5)
ollama --version

# Start the service
sudo systemctl enable ollama
sudo systemctl start ollama
sudo systemctl status ollama  # should show: active (running)

Configure Ollama as Systemd Service with GPU Override

To make sure the HSA override and performance settings are always applied when Ollama starts as a service:

Edit Ollama service environmentbash
sudo mkdir -p /etc/systemd/system/ollama.service.d
sudo nano /etc/systemd/system/ollama.service.d/override.conf

Paste this into the file:

/etc/systemd/system/ollama.service.d/override.confini
[Service]
Environment="HSA_OVERRIDE_GFX_VERSION=10.3.0"
Environment="OLLAMA_FLASH_ATTENTION=1"
Environment="OLLAMA_MAX_LOADED_MODELS=1"
Environment="OLLAMA_CONTEXT_LENGTH=16384"
Apply and restartbash
sudo systemctl daemon-reload
sudo systemctl restart ollama

# Test the API
curl http://localhost:11434/api/tags

Open WebUI — Browser Frontend

Open WebUI gives you a full ChatGPT-like interface in your browser. It connects to Ollama, stores conversation history, supports file uploads, RAG over documents, and lets you paste code and ask questions about it.

Install with Docker (Recommended)

Install Docker first if neededbash
sudo apt install -y docker.io docker-compose
sudo systemctl enable --now docker
sudo usermod -aG docker $USER
# Log out and back in for group to apply
Run Open WebUI (connects to your existing Ollama)bash
docker run -d \
  --name open-webui \
  --restart unless-stopped \
  -p 3000:8080 \
  -v open-webui:/app/backend/data \
  -e OLLAMA_BASE_URL=http://host.docker.internal:11434 \
  --add-host=host.docker.internal:host-gateway \
  ghcr.io/open-webui/open-webui:main

Open http://localhost:3000 in your browser. Create an admin account on first launch.

💡
No Docker? You can also install directly: pip install open-webui && open-webui serve

Open WebUI — Key Features for Coding

📁 File Upload

Upload your .py / .js / .ts files directly into the chat. Ask the model to review, refactor, or explain them.

📚 Knowledge Base (RAG)

Add your entire codebase as a document set. The model can search your code to answer architecture questions.

🔀 Model Switcher

Switch between qwen3.5:9b and qwen3.5:2b instantly in the UI. Use the small one for quick questions.

💬 Conversation History

Full persistent history. Reference old conversations, branch them, or share them.

Aider — Claude Code-Style Terminal Agent

Aider is the closest open-source equivalent to Claude Code. It runs in your terminal, reads your entire git repo, makes multi-file edits, and can run commands. This is your primary coding agent.

Install Aider

Installbash
# Install with pip
pip install aider-chat --break-system-packages

# OR via pipx (cleaner, isolated)
pipx install aider-chat

# Verify
aider --version

Configure Aider for Ollama + Qwen3.5

Create ~/.aider.conf.yml (global config)yaml
# ~/.aider.conf.yml
model: ollama/qwen3.5:9b
edit-format: diff          # Use diffs for clean edits
auto-commits: true         # Auto git commit each change
dirty-commits: true        # Allow commits on dirty repos
stream: true               # Stream output as it generates
no-check-model-accepts-settings: true

Run Aider

Basic usagebash
# Navigate to your project
cd ~/myproject

# Start Aider with Qwen3.5 via Ollama
aider --model ollama/qwen3.5:9b

# Or with specific files to edit
aider --model ollama/qwen3.5:9b app.py utils.py

# Or let Aider read all files in the repo
aider --model ollama/qwen3.5:9b --map-tokens 2048

Aider Commands You'll Use Daily

CommandWhat it does
/add filename.pyAdd a file to the edit context

Continue.dev — VS Code Integration

Continue turns VS Code into a local Cursor/Copilot. You get inline chat, tab autocomplete, code editing from the sidebar, and full codebase context — all routing through your local Ollama.

Install Continue Extension

In VS Code: Extensions (Ctrl+Shift+X) → Search "Continue" → Install. Or from terminal:

Install via VS Code CLIbash
code --install-extension Continue.continue

Configure Continue for Ollama + Qwen3.5

Open the Continue sidebar (click the Continue icon in VS Code) → Gear icon → Open config.yaml. Replace the content with:

~/.continue/config.yamlyaml
name: Local Qwen3.5
version: 1.0.0
schema: v1

models:
  # Main chat / edit model
  - name: Qwen3.5 9B
    provider: ollama
    model: qwen3.5:9b
    roles:
      - chat
      - edit
      - apply
      - summarize

  # Fast autocomplete model
  - name: Qwen3.5 2B (autocomplete)
    provider: ollama
    model: qwen3.5:2b
    roles:
      - autocomplete

  # Embeddings for codebase search
  - name: nomic-embed-text
    provider: ollama
    model: nomic-embed-text
    roles:
      - embed

# Context providers — what the model can see
context:
  - provider: code
  - provider: docs
  - provider: diff
  - provider: terminal
  - provider: problems
  - provider: folder
  - provider: codebase

How to Chat & Ask It to Edit Your Code

Qwen3.5 has a built-in thinking mode — before answering, it reasons through the problem inside <think> tags. This makes it significantly better at complex coding tasks. Here's how to use each interface effectively:

In Open WebUI (Browser)

Best for: exploring code, asking questions, reviewing files you paste in.

Explain what this function does and what edge cases it misses: [paste code here]
Refactor this Python class to use dataclasses and add type hints: [paste code]
I'm getting this error: [paste error]. Here's my code: [paste code]. What's wrong and how do I fix it?
Write unit tests for this function using pytest: [paste function]

In Aider (Terminal Agent)

Best for: actually editing files in your project. Aider reads your repo and makes real changes.

Example Aider sessionbash
# Start in your project
cd ~/myproject
aider --model ollama/qwen3.5:9b

# Aider prompt (>) — example requests:

> Add error handling to the database connection in db.py

> Refactor the UserService class to follow SOLID principles

> There's a bug in api.py — the /users endpoint returns 500 when
  the user doesn't exist. Find and fix it.

> /add tests/test_api.py
> Write comprehensive pytest tests for the UserService class

> /run pytest
# Aider sees the failures and fixes them automatically

In Continue.dev (VS Code)

Best for: inline edits while you code, asking questions about what's on screen.

Inline Edit (Ctrl+I)

Select a function → press Ctrl+I → type what you want changed → it rewrites inline with a diff preview you can accept/reject.

Chat Sidebar (Ctrl+L)

Ask questions with full file context. Use @filename to reference specific files in your question.

Autocomplete (Tab)

As you type, the 2B model suggests completions. Much faster than the 9B, good enough for boilerplate.

Prompt Templates That Work Well

TaskPrompt Pattern
Fix a bug "The function X does Y but it should do Z. Here's the error: [error]. Fix it."
Add feature "Add [feature] to [file]. It should [behavior]. Don't break existing functionality."
Refactor "Refactor [class/function] to be more readable. Keep the same public API."
Write tests "Write pytest tests for [function]. Cover happy path, edge cases, and error cases."
Code review "Review this code for bugs, security issues, and performance problems. Be specific."
Explain "Explain how [code] works step by step. I'm familiar with Python but new to this library."
💡
Disable thinking mode for quick tasks: Add /no_think at the end of your prompt when you just need a fast answer and don't need deep reasoning. Example: What does enumerate() do? /no_think

Performance Tuning

Optimize Ollama for Your Setup

Add to your systemd override or .bashrcbash
# Tune context length (lower = faster, less RAM, fewer tokens per request)
export OLLAMA_CONTEXT_LENGTH=8192   # 8K is good for most coding tasks

# Flash attention — reduces memory bandwidth ~40% on supported hardware
export OLLAMA_FLASH_ATTENTION=1

# Only load 1 model at a time (saves VRAM)
export OLLAMA_MAX_LOADED_MODELS=1

# Keep model in RAM between requests (faster response on next call)
# Default is 5m; set to -1 to keep forever, 0 to unload immediately
export OLLAMA_KEEP_ALIVE=10m

Try Vulkan if ROCm Gives Problems

If your RX 6400 still runs on CPU despite the HSA override, Vulkan is an experimental but functional alternative:

Enable Vulkan backendbash
export OLLAMA_VULKAN=1
sudo systemctl restart ollama

# Install Vulkan if needed
sudo apt install -y mesa-vulkan-drivers vulkan-tools
vulkaninfo | grep "GPU id"

Monitor GPU Usage

Real-time GPU monitoringbash
# Watch VRAM and GPU utilization
watch -n 1 rocm-smi

# Or install amdgpu_top (Rust-based, better UI)
sudo apt install amdgpu-top
amdgpu_top

# Check if Ollama is using GPU layers
curl http://localhost:11434/api/ps

Troubleshooting

ProblemFix
ollama: model not found for qwen3.5 Update Ollama: curl -fsSL https://ollama.com/install.sh | sh — Qwen3.5 requires v0.17+
GPU not detected / running on CPU only Verify HSA_OVERRIDE_GFX_VERSION=10.3.0 is in the systemd override. Check rocm-smi detects card.
Model crashes / OOM error Reduce context: set OLLAMA_CONTEXT_LENGTH=4096. Or use qwen3.5:4b instead of 9b.
Aider can't connect to Ollama Make sure Ollama is running: systemctl status ollama. Test API: curl http://localhost:11434/api/tags
Continue.dev autocomplete not working Ensure qwen3.5:2b is pulled and listed under roles: - autocomplete in config.yaml
Open WebUI can't reach Ollama in Docker Make sure you have --add-host=host.docker.internal:host-gateway in your docker run command (required on Linux)
Qwen3.5 repeats itself Re-pull the model: ollama pull qwen3.5:9b — Ollama patched this bug (presence penalty fix)
Slow generation (<1 tok/s) Model fully on CPU. Check GPU detection. Try OLLAMA_VULKAN=1 as alternative backend.
💡
Check Ollama logs in real time:
journalctl -u ollama -f — Look for lines mentioning n_gpu_layers. If it shows 0, no GPU offload is happening.
